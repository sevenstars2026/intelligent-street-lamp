## 一、自动控制功能

### 1.1 旧方案（已实现 ✅）— PR #10

`device-control.service.ts` 中的 `TODO` 已被移除，替换为 `evaluateAutoControl()` 方法：

- 接收硬件光照数据后，检测设备是否为 `auto` 模式
- 查询阈值 → `lightIntensity < thresholdOn` 则开灯，`lightIntensity > thresholdOff` 则关灯
- `autoControlInFlight` 集合防重入
- operator 设为 `operatorId: 0, operatorName: '自动控制'`

**局限性：** 此方案由后端做决策，依赖 MQTT 实时连接。设备离线时无法自主控制。

### 1.2 新方案（仍未实现 ❌）— 自动控制功能接口规范 v1.1

📄 **设计文档：** `自动控制功能接口规范.md`（v1.1，状态标注为"待实现"）

新方案将自动判断职责移至**硬件本地**，后端只负责配置下发和事后记录。这是更健壮的 IoT 架构。

#### 后端 P0（3 项，必须实现）

| # | 功能 | 描述 |
|---|------|------|
| 1 | **API 变更后推送 config** | 用户调用 `POST /threshold` 或 `PUT /mode` 后，通过 MQTT 向硬件推送 `devices/{id}/config`（含 mode、thresholdOn、thresholdOff、version） |
| 2 | **响应 config/request** | 订阅 `devices/+/config/request`，硬件上电/重连时查询 DB 最新配置并返回 |
| 3 | **接收 auto-action 事件** | 订阅 `devices/+/auto-action`，接收硬件自主开关灯事件，更新 `devices.current_state` 并写入 `control_logs`（operator_id=0, operator_name='自动控制'） |

#### 后端 P1（2 项，应该实现）

| # | 功能 | 描述 |
|---|------|------|
| 4 | **config 版本号管理** | 每次推送 config 时 version 单调递增（可用 `updated_at` 毫秒时间戳），硬件据此判断是否更新 |
| 5 | **config/ack 处理** | 订阅 `devices/+/config/ack`，记录硬件是否成功接受配置 |

#### 新增 MQTT Topic（3 个，均未订阅）

| Topic | 方向 | 用途 |
|-------|------|------|
| `devices/{id}/config` | 后端 → 硬件 | 下发模式和阈值配置 |
| `devices/{id}/config/request` | 硬件 → 后端 | 硬件上电/重连时请求配置 |
| `devices/{id}/auto-action` | 硬件 → 后端 | 硬件上报自主开关灯事件 |

#### 兼容性声明（来自设计文档）

- **REST API：零影响。** 11 个现有 API 接口不变
- **数据库：零 DDL。** 不建新表、不改字段，所有自动控制数据写入现有表
- **前端：零影响。** 控制日志中 `operator_name === '自动控制'` 即可区分来源
- **MQTT：新增 3 个 topic**，不影响现有 4 个 topic 的行为
