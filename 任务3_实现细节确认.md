  - 选项A：30秒没收到心跳算离线
  - 选项B：60秒没收到心跳算离线
  - 选项C：120秒（2分钟）没收到心跳算离线
  - 选项D：根据心跳频率动态计算（比如：心跳频率 × 3）

**回答：**
> 选项D：根据心跳频率动态计算（心跳频率 × 3）
> 
> 理由：灵活适应不同的心跳频率配置



**问题：**
- 离线检测如何实现？
  - 选项A：定时任务每10秒扫描数据库，检查last_heartbeat_time
  - 选项B：Redis存储心跳时间，设置TTL，过期触发告警
  - 选项C：收到心跳时更新Redis TTL，单独的定时任务检查Redis key是否存在

**回答：**
> 选项C：收到心跳时更新Redis TTL + 定时任务检查
> 
> 实现方案：
> 1. 收到心跳时：SET device:heartbeat:{deviceId} {timestamp} EX {心跳频率×3}
> 2. 定时任务每30秒检查：遍历所有设备，检查Redis key是否存在
> 3. key不存在 → 设备离线 → 触发告警
> 4. 告警去重：已告警设备记录在 device:offline:alarm:{deviceId}

### Q5: 告警服务 - 告警频率控制

**背景：** 同一设备可能持续离线，避免重复告警轰炸。

**问题：**
- 同一设备离线后，如何控制告警频率？
  - 选项A：只告警一次，直到设备恢复后再次离线才重新告警
  - 选项B：每小时提醒一次（持续离线的情况）
  - 选项C：第一次立即告警，之后按时间升级（1小时后提醒、6小时后提醒、24小时后提醒）
  - 选项D：只告警一次，不重复提醒

**回答：**
> 选项C：第一次立即告警，之后按时间升级
> 
> 实现方案：
> 1. 设备离线时立即发送告警
> 2. 记录告警时间：device:offline:alarm:{deviceId} 存储初次告警时间
> 3. 定时任务检查：如果离线时长达到1小时、6小时、24小时，再次发送告警
> 4. 每次告警升级时更新告警记录的等级



**问题：**
- 设备恢复上线后，是否需要通知？
  - 选项A：需要通知"设备已恢复"
  - 选项B：不需要通知，只记录日志
  - 选项C：只有告警过的设备恢复时才通知

**回答：**
> 选项A：需要通知"设备已恢复"
> 
> 实现方案：
> 1. 检测到设备心跳恢复时，发送"设备已恢复"通知
> 2. 清除 device:offline:alarm:{deviceId} 标记
> 3. 更新告警记录状态为"已恢复"

### Q10: 错误处理和边界情况

> 
> 响应示例：
> ```json
> {

>   "message": "设备不存在",
>   "data": null
> }

**问题：**
- 如果MQTT服务不可用，控制指令如何处理？
  - 选项A：立即返回错误"服务不可用"
  - 选项B：将指令放入队列，等待MQTT恢复后重试
  - 选项C：记录日志，返回错误，不重试

**回答：**
> 选项C：记录日志，返回错误，不重试
> 
> 实现方案：
> 1. 捕获MQTT发送失败异常
> 2. 记录错误日志（设备ID、指令、时间、错误堆栈）
> 3. 返回503错误
> 4. 如果MQTT连接断开，触发运维告警
> 
> 响应示例：
> ```json
> {
>   "code": 503,
>   "message": "控制服务暂时不可用，请稍后重试",
>   "data": null
> }
> ```
> 
> 前端可提供"重试"按钮让用户手动重试



**问题：**
- 如果数据库查询超时或失败，如何处理？
  - 选项A：返回500错误，记录日志
  - 选项B：返回500错误，记录日志，触发告警通知运维
  - 选项C：重试3次，失败后返回错误

**回答：**
> 选项C：重试3次，失败后返回错误
> 
> 实现方案：
> 1. 捕获数据库查询异常
> 2. 快速重试3次（间隔：100ms, 200ms, 500ms）
> 3. 3次都失败后：
>    - 记录错误日志（SQL、参数、堆栈）
>    - 返回500错误
>    - 如果连续失败次数过多，触发告警
> 
> 响应示例：
> ```json
> {
>   "code": 500,
>   "message": "服务器内部错误，请稍后重试",
>   "data": null
> }
> ```
> 
> 注意：写操作（INSERT/UPDATE）需要保证幂等性，避免重试导致重复写入

## 📊 实现细节总结

### 1. 自动化规则引擎
- **触发逻辑**：最近5次采样中有4次低于/高于阈值才触发
- **模式管理**：设备支持自动/手动模式切换，存储在数据库
- **防抖动**：避免频繁开关

### 2. 设备控制服务
- **控制类型**：开/关（on/off）
- **权限**：市政人员和管理员
- **反馈机制**：等待硬件反馈，10秒超时
- **批量控制**：支持，返回每个设备的执行结果

### 3. 告警服务
- **离线判定**：心跳频率 × 3
- **检测方式**：Redis TTL + 定时任务
- **通知方式**：站内消息 + 邮件（策略模式设计，便于扩展）
- **通知对象**：所有管理员
- **告警频率**：首次立即告警，之后1小时、6小时、24小时升级
- **恢复通知**：需要通知"设备已恢复"
- **告警等级**：按离线时长分级（低/中/高）
- **告警类型**：设备离线、控制失败、频繁开关

### 4. 数据统计服务
- **时间范围**：固定范围（24h/7d/30d）+ 自定义
- **数据保留**：原始数据1个月，聚合数据1年
- **聚合维度**：按小时（平均/最大/最小值）
- **聚合方式**：混合模式（最近24h实时，历史数据预计算）

### 5. API设计
- **命名风格**：RESTful
- **返回格式**：遵循任务2规范（建议统一包装格式）
- **错误处理**：
  - 设备不存在：404
  - MQTT不可用：503，记录日志，不重试
  - 数据库失败：重试3次，失败返回500
