# 景区游客端 - 后端功能设计（修正版）

> 基于《景区游客端-计划书.md》和《景区游客端-后端功能设计.md》校正。  
> 日期：2026-07-08  
> 状态：可实施设计  
> 实现信心：95%（已对照当前 `task3-backend` 表结构、类型定义、路由注册、告警页展示逻辑）

---

## 1. 设计结论

游客端后端扩展可以落地，但原设计不能原样实现。必须修正以下点：

1. `alarms.alarm_type` 当前只支持 `offline`、`control_failed`、`frequent_switch`，必须扩展 `fault_report`。
2. 游客上报图片不适合只塞进 `alarms.message`，MVP 可写 message，但推荐新增 `fault_reports` 表保存游客信息和图片路径。
3. 上传字段统一为 `photos`，后端使用 `upload.array('photos', 3)`。
4. 上传目录需要开放静态访问：`/uploads/...`。
5. 景区静态路灯数据第一版必须只引用当前已有设备：`lamp_001`、`lamp_002`、`lamp_003`。
6. 管理端告警页需要补充 `fault_report: '游客上报'` 映射。

以上修正后，功能实现和现有系统不冲突。

## 6. 数据库设计修正

### 6.1 告警类型扩展

当前建表 SQL 需要从：

```sql
alarm_type ENUM('offline', 'control_failed', 'frequent_switch') NOT NULL
```

改为：

```sql
alarm_type ENUM('offline', 'control_failed', 'frequent_switch', 'fault_report') NOT NULL
```

真实 MySQL 已存在表时需要迁移：

```sql
ALTER TABLE alarms
MODIFY alarm_type ENUM('offline', 'control_failed', 'frequent_switch', 'fault_report') NOT NULL;
```

### 6.2 新增 fault_reports 表

推荐新增表保存游客信息和图片路径：

```sql
CREATE TABLE IF NOT EXISTS fault_reports (
  id INT AUTO_INCREMENT PRIMARY KEY,
  alarm_id INT NOT NULL,
  reporter_name VARCHAR(100) NOT NULL,
  reporter_phone VARCHAR(30) NOT NULL,
  lamp_id VARCHAR(64) NOT NULL,
  description TEXT NOT NULL,
  photo_urls JSON NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_fault_reports_alarm_id (alarm_id),
  INDEX idx_fault_reports_lamp_created (lamp_id, created_at)
);
```

`alarms` 保持告警主记录，`fault_reports` 保存游客上报详情。这比把所有内容塞进 `message` 更稳，也方便后续管理端展示图片。

## 7. 故障上报接口

### 7.1 请求

`POST /api/reports`

`Content-Type: multipart/form-data`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| reporterName | text | 是 | 上报人姓名 |
| reporterPhone | text | 是 | 手机号，仅记录，不验证短信 |
| lampId | text | 是 | 路灯编号 |
| description | text | 是 | 故障描述 |
| photos | file[] | 否 | 最多 3 张，每张不超过 5MB |

### 7.2 校验规则

- `reporterName`、`reporterPhone`、`lampId`、`description` 必填。
- `lampId` 必须存在于 `devices` 表或静态 `lampPositions` 中。推荐优先查 `DatabaseService.getDevice(lampId)`。
- 图片格式只允许 `image/jpeg`、`image/png`、`image/webp`。
- 图片最多 3 张。
- 单张图片最大 5MB。

### 7.3 写入流程

```text
1. multer 保存图片到 uploads/reports/
2. 校验表单字段
3. 查询设备，获取 deviceName
4. DatabaseService.addAlarm({
     deviceId: lampId,
     deviceName,
     alarmType: 'fault_report',
     alarmLevel: 'medium',
     status: 'active',
     message: `游客 ${name}(${phone}) 上报故障: ${description}`,
     createdAt: new Date(),
     handledAt: null,
     handlerId: null,
     handlerName: null
   })
5. 写入 fault_reports，保存 alarm_id、游客信息、photo_urls
6. 返回 alarmId/reportId
```

### 7.4 响应

成功：

{
  "code": 200,

## 13. 风险和处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 真实 MySQL enum 未迁移 | `fault_report` 插入失败 | 实现时必须执行 `ALTER TABLE` 或在初始化中检测修正 |
| 上传目录不存在 | 图片保存失败 | multer destination 前创建目录 |
| 字段名不一致 | 前端传图失败 | 统一为 `photos` |
| 静态 lampId 不存在 | 上报无法关联设备 | 第一版只使用 `lamp_001` 到 `lamp_003` |
| 管理端无中文映射 | 展示不友好 | 增加 `fault_report: '游客上报'` |

## 14. 95% 信心判断

达到 95% 实现信心的原因：

1. 景区数据接口是静态 GET，不依赖数据库，不会冲击现有业务。
2. 故障上报复用现有告警查询和处理流程，只扩展告警类型。
3. 新增 `fault_reports` 表避免污染现有 `alarms` 表结构。
4. 上传能力由成熟中间件 `multer` 提供，风险可控。
5. 已明确当前最大冲突点：MySQL enum、TS union、管理端映射、上传字段名。

暂不达到 100% 的原因：

1. 真实服务器 MySQL 当前 enum 是否可自动迁移，需要实际执行后确认。
2. 游客端尚未实现，最终字段可能需要和游客端表单再对齐。

结论：按本文档实施，预计可通过后端接口测试和管理端告警联动测试。
