## 1. 功能概述

### 1.1 核心能力

告警系统负责**自动检测异常 → 生成告警 → 升级告警 → 人工处理**的完整闭环：

| 环节 | 触发方式 | 说明 |
|------|----------|------|
| 告警生成 | 自动（定时任务 / 控制失败回调） | 系统自动检测异常并生成告警记录 |
| 告警升级 | 自动（定时任务） | 未处理告警随时间自动提升级别 |
| 告警查询 | 用户主动 | 支持按状态、设备、类型、级别筛选 |
| 告警处理 | 用户主动 | 管理员标记告警为"已处理"，填写备注 |

### 1.2 告警类型

| 类型 | 标识 | 触发条件 | 默认级别 |
|------|------|----------|----------|
| 设备离线 | `offline` | 设备超过 5 分钟未上报心跳 | `high` |
| 控制失败 | `control_failed` | 向设备下发控制指令后返回失败 | `medium` |
| 频繁开关 | `frequent_switch` | 同一设备 10 分钟内开关超过 5 次 | `low` |

### 1.3 告警级别

| 级别 | 标识 | 含义 | 前端颜色 |
|------|------|------|----------|
| 低级 | `low` | 需关注，暂无影响 | 蓝色 |
| 中级 | `medium` | 可能影响正常运行 | 橙色 |
| 高级 | `high` | 已影响设备运行 | 红色 |
| 紧急 | `critical` | 需立即处理（由升级算法自动提升） | 深红色 |

### 1.4 与现有系统的关系

- ✅ **数据库 `alarms` 表已存在**，DatabaseService 中已实现完整 CRUD（增删改查），无需重新设计数据层
- ✅ **告警类型定义已存在**（`database.types.ts` 中的 `Alarm` 接口）
- ❌ **REST 路由层缺失**：需新增 Controller + Routes
- ❌ **定时任务缺失**：scheduler.service.ts 已删除，需重新实现
- ❌ **控制失败时的告警生成逻辑缺失**：需在 control 流程中插入告警创建
- ❌ **前端告警页面为空占位**：需实现完整 UI

## 4. 告警触发规则

### 4.1 设备离线告警（offline）

**触发方式：** 定时任务，每分钟执行一次

**触发逻辑：**
1. 查询 `devices` 表中所有设备
2. 检查每个设备的 `last_heartbeat` 字段
3. 若 `NOW() - last_heartbeat > 5分钟`，且设备 `status` 仍为 `'online'`：
   - 生成离线告警（level = `high`）
   - 更新设备 `status` 为 `'offline'`
4. 若设备已离线且已有 `active` 状态的离线告警，则**不重复生成**

**去重规则：** 同一设备在已有未处理的离线告警时，不重复生成。

**告警消息模板：**
```
设备 {deviceName}({deviceId}) 已离线超过5分钟，最后心跳时间 {lastHeartbeat}
```

### 4.2 控制失败告警（control_failed）

**触发方式：** 实时触发（在控制流程中同步生成）

**触发逻辑：**
在 `DeviceControlService.controlDevice()` 方法中，当控制结果为 `failed` 或 `timeout` 时：
1. 调用 `DatabaseService.addAlarm()` 生成告警
2. 不影响原有控制流程（告警生成失败不应阻塞控制响应返回）

**告警消息模板：**
- 控制失败：`设备 {deviceName}({deviceId}) 控制失败：{resultMessage}`
- 控制超时：`设备 {deviceName}({deviceId}) 控制超时（10秒未响应），请检查设备状态`

### 4.3 频繁开关告警（frequent_switch）

**触发方式：** 实时触发（在控制日志写入后检查）

**触发逻辑：**
每次向 `control_logs` 写入一条成功后，检查该设备近 10 分钟内的控制记录：
1. 查询 `control_logs` 表，条件：`device_id = ? AND request_time > NOW() - INTERVAL 10 MINUTE`
2. 若记录数 ≥ 5，生成频繁开关告警（level = `low`）

**去重规则：** 同一设备在已有未处理的频繁开关告警时，不重复生成。

**告警消息模板：**
```
设备 {deviceName}({deviceId}) 近10分钟内开关操作达{N}次，触发频繁开关告警
```

## 5. 告警升级算法

### 5.1 升级规则

| 条件 | 升级规则 |
|------|---------|
| 告警创建后 **24 小时**未处理 | `low` → `medium` |
| 告警创建后 **48 小时**未处理 | `medium` → `high` |
| 告警创建后 **72 小时**未处理 | `high` → `critical` |

> 注：原数据模型中 `alarmLevel` 只有 `low/medium/high` 三级。建议将 `Alarm` 类型扩展增加 `critical` 级别，或在此阶段将 `high` 作为最高级别。

### 5.2 升级执行

**触发方式：** 定时任务，每小时执行一次

**伪代码：**

```typescript
async function upgradeAlarms() {
  const activeAlarms = await DatabaseService.getAlarms({ status: 'active' });
  const now = new Date();

  for (const alarm of activeAlarms) {
    const hoursSinceCreation = (now.getTime() - alarm.createdAt.getTime()) / (1000 * 3600);
    let newLevel: string | null = null;

    if (hoursSinceCreation >= 72 && alarm.alarmLevel !== 'high') {
      newLevel = 'high';
    } else if (hoursSinceCreation >= 48 && alarm.alarmLevel === 'low') {
      newLevel = 'high';
    } else if (hoursSinceCreation >= 24 && alarm.alarmLevel === 'low') {
      newLevel = 'medium';
    }

    if (newLevel) {
      const message = `[自动升级] ${alarm.message}；已持续${Math.floor(hoursSinceCreation)}小时未处理，级别由 ${alarm.alarmLevel} 升至 ${newLevel}`;
      await DatabaseService.updateAlarmLevel(alarm.id, newLevel as any, message);
      console.log(`[AlarmUpgrade] ⬆️ alarm #${alarm.id} ${alarm.alarmLevel} → ${newLevel}`);
    }
  }
}
```

### 7.3 告警控制器实现参考

```typescript
// src/controllers/alarm.controller.ts

import { Request, Response } from 'express';
import { AlarmService } from '../services/alarm.service';

export class AlarmController {
  
  // GET /api/alarms
  static async getAlarms(req: Request, res: Response) {
    try {
      const { status, deviceId, alarmType, alarmLevel, page = '1', pageSize = '20' } = req.query;
      
      const pageNum = Math.max(1, parseInt(page as string, 10) || 1);
      const size = Math.min(100, Math.max(1, parseInt(pageSize as string, 10) || 20));
      
      const result = await AlarmService.getAlarms(
        { status, deviceId, alarmType, alarmLevel } as any,
        pageNum,
        size
      );
      
      res.json({ code: 200, message: 'success', data: result });
    } catch (error) {
      console.error('[AlarmController] getAlarms error:', error);
      res.status(500).json({ code: 500, message: '服务器内部错误', data: null });
    }
  }
  
  // GET /api/alarms/:alarmId
  static async getAlarm(req: Request, res: Response) {
    try {
      const alarmId = parseInt(req.params.alarmId);
      if (isNaN(alarmId)) {
        return res.status(400).json({ code: 400, message: '告警ID无效', data: null });
      }
      
      const alarm = await AlarmService.getAlarm(alarmId);
      if (!alarm) {
        return res.status(404).json({ code: 404, message: '告警不存在', data: null });
      }
      
      res.json({ code: 200, message: 'success', data: alarm });
    } catch (error) {
      console.error('[AlarmController] getAlarm error:', error);
      res.status(500).json({ code: 500, message: '服务器内部错误', data: null });
    }
  }
  
  // PUT /api/alarms/:alarmId/resolve
  static async resolveAlarm(req: Request, res: Response) {
    try {
      const alarmId = parseInt(req.params.alarmId);
      if (isNaN(alarmId)) {
        return res.status(400).json({ code: 400, message: '告警ID无效', data: null });
      }
      
      const user = (req as any).user;  // 从 mock JWT 中间件获取
      const { note } = req.body;
      
      const result = await AlarmService.resolveAlarm(alarmId, user.id, user.name, note);
      
      res.json({ code: 200, message: '告警已处理', data: result });
    } catch (error: any) {
      if (error.message === 'ALARM_NOT_FOUND') {
        return res.status(404).json({ code: 404, message: '告警不存在', data: null });
      }
      if (error.message === 'ALARM_ALREADY_RESOLVED') {
        return res.status(400).json({ code: 400, message: '该告警已处理', data: null });
      }
      console.error('[AlarmController] resolveAlarm error:', error);
      res.status(500).json({ code: 500, message: '服务器内部错误', data: null });
    }
  }
}
```

### 8.4 仪表盘告警提醒条

## 9. 数据库 DDL

### 9.1 建表语句（若表不存在）

```sql
CREATE TABLE IF NOT EXISTS alarms (
  id INT AUTO_INCREMENT PRIMARY KEY COMMENT '主键ID',
  device_id VARCHAR(50) NOT NULL COMMENT '设备ID',
  device_name VARCHAR(100) NOT NULL COMMENT '设备名称',
  alarm_type VARCHAR(30) NOT NULL COMMENT '告警类型：offline/control_failed/frequent_switch',
  alarm_level VARCHAR(10) NOT NULL DEFAULT 'low' COMMENT '告警级别：low/medium/high',
  status VARCHAR(10) NOT NULL DEFAULT 'active' COMMENT '告警状态：active/resolved',
  message TEXT NOT NULL COMMENT '告警详细描述',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  handled_at DATETIME NULL COMMENT '处理时间',
  handler_id INT NULL COMMENT '处理人ID',
  handler_name VARCHAR(50) NULL COMMENT '处理人姓名',
  
  INDEX idx_device_id (device_id),
  INDEX idx_alarm_type (alarm_type),
  INDEX idx_alarm_level (alarm_level),
  INDEX idx_status (status),
  INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警记录表';
```

### 9.2 数据清理 SQL（配合定时任务）

```sql
-- 删除 7 天前已处理的告警
DELETE FROM alarms 
WHERE status = 'resolved' 
  AND handled_at < NOW() - INTERVAL 7 DAY;
```

## 11. MQTT 协议参考

告警系统涉及以下现有 MQTT Topic（无需新增）：

| Topic | 方向 | 与告警的关系 |
|-------|------|-------------|
| `devices/{id}/heartbeat` | 硬件→后端 | 心跳超时触发离线告警 |
| `devices/{id}/control` | 后端→硬件 | 控制响应超时/失败触发控制告警 |
| `devices/{id}/control/response` | 硬件→后端 | 控制失败触发控制告警 |

**告警系统不引入新的 MQTT Topic。**
