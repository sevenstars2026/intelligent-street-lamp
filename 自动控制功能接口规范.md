## 概述

自动控制功能的核心场景：**当设备与后端断开连接（离线）时，硬件仍能根据本地存储的光照阈值自主开关灯。** 即使设备在线，自动控制的判断也由硬件本地完成，后端只负责配置下发和事后记录。

### 角色分工

| 职责 | 后端 | 硬件 |
|------|:----:|:----:|
| 存储和管理阈值配置 | ✅ | — |
| 存储和管理控制模式（auto/manual） | ✅ | — |
| 将配置同步到硬件 | ✅ | — |
| 接收并记录自动控制事件 | ✅ | — |
| 手动控制指令下发 | ✅ | — |
| **本地光照检测** | — | ✅ |
| **本地阈值比对** | — | ✅ |
| **本地开关灯执行** | — | ✅ |
| 上报自动控制动作 | — | ✅ |
| 上报心跳和光照数据 | — | ✅ |

## 1. 架构

```
┌─────────────────────────────────────────────────┐
│                     后端                         │
│                                                  │
│  用户设置阈值/模式 ──▶ 写入 DB ──▶ MQTT 推送配置  │
│                                                  │
│  MQTT 接收 auto-action ──▶ 更新状态 + 写日志     │
│                                                  │
│  用户手动控制 ──▶ MQTT 下发 control 指令         │
└──────────────────────┬──────────────────────────┘
                       │
          ┌────────────┼────────────┐
          │  config     │  control   │  auto-action
          ▼            ▼            ▲
┌─────────────────────────────────────────────────┐
│                     硬件                         │
│                                                  │
│  启动/重连 ──▶ 请求配置 ──▶ 存入本地存储         │
│                                                  │
│  定时读取光照传感器                               │
│    ├─ mode=auto  → 本地比对阈值 → 自主开关灯      │
│    │               → 发布 auto-action 上报后端    │
│    └─ mode=manual → 仅等待后端 control 指令       │
│                                                  │
│  收到 control 指令 → 执行 → 回复 response         │
└─────────────────────────────────────────────────┘
```

**关键原则：**
- 硬件本地存储一份阈值和模式配置，**离线时仍可正常工作**
- 后端变更配置时 **主动推送** 到硬件，硬件立即更新本地配置
- 硬件上电/重连时 **主动拉取** 最新配置

## 2. 数据库约束

不新增表。现有 `devices` 和 `thresholds` 表作为配置的权威数据源（Source of Truth）。

### 2.1 devices 表

| 列 | 类型 | 说明 |
|----|------|------|
| `id` | varchar(50) | `lamp_` 前缀 |
| `mode` | enum('auto','manual') | 由用户通过 API 设置，后端负责同步到硬件 |
| `current_state` | enum('on','off') | 硬件上报 auto-action 事件时更新 |

### 2.2 thresholds 表

| 列 | 类型 | 说明 |
|----|------|------|
| `device_id` | varchar(50) | PK |
| `light_threshold_on` | int | 光照低于此值 → **硬件本地**开灯 |
| `light_threshold_off` | int | 光照高于此值 → **硬件本地**关灯 |

### 2.3 阈值语义（运行在硬件上）

```
光照 (lux)
  ▲
  │  ┌───────────────────  daylight：不动作
  │  │
  ├──┼── thresholdOff ──── 光照高于此值 → 关灯
  │  │
  │  │   滞回区间（hysteresis）：不动作，防止抖动
  │  │
  ├──┼── thresholdOn ───── 光照低于此值 → 开灯
  │  │
  │  └───────────────────  night：不动作
  └──────────────────────────────→
```

| 条件 | 硬件动作 |
|------|---------|
| 光照 **<** `thresholdOn` 且当前灯灭 | 开灯 |
| 光照 **>** `thresholdOff` 且当前灯亮 | 关灯 |
| `thresholdOn` ≤ 光照 ≤ `thresholdOff` | 不动作 |
| 灯光状态已符合阈值方向 | 不动作（不重复开关） |

### 2.4 control_logs 表

硬件自主触发的开关动作，由后端收到 `auto-action` 事件后写入此表：

| 列 | 自动控制写入值 |
|----|--------------|
| `device_id` | 设备 ID |
| `command` | `'on'` / `'off'` |
| `status` | `'success'`（硬件已执行完毕） |
| `operator_id` | **`0`** |
| `operator_name` | **`'自动控制'`** |
| `request_time` | 硬件执行时间 |
| `response_time` | 同上（无需等待） |
| `result_message` | 触发原因，如 `"光照 180 < 阈值 200"` |

## 3. MQTT 协议约束

自动控制功能需要 **新增 3 个 topic**：

| Topic | 方向 | 新增/现有 | 用途 |
|-------|------|:---:|------|
| `devices/{id}/config` | 后端→硬件 | **新增** | 下发模式+阈值配置 |
| `devices/{id}/config/request` | 硬件→后端 | **新增** | 硬件上电/重连时请求配置 |
| `devices/{id}/auto-action` | 硬件→后端 | **新增** | 硬件上报自主开关灯事件 |
| `devices/{id}/data` | 硬件→后端 | 现有 | 光照数据上报 |
| `devices/{id}/heartbeat` | 硬件→后端 | 现有 | 心跳 |
| `devices/{id}/control` | 后端→硬件 | 现有 | 手动控制指令 |
| `devices/{id}/control/response` | 硬件→后端 | 现有 | 手动控制响应 |

### 3.1 配置下发 `devices/{id}/config`

```
后端 → 硬件  QOS 1（至少一次送达）
触发时机：用户通过 API 修改模式或阈值后立即推送
```

```json
{
  "mode": "auto",
  "thresholdOn": 200,
  "thresholdOff": 600,
  "version": 3,
  "timestamp": 1751436000000
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `mode` | string | ✅ | `"auto"` / `"manual"` |
| `thresholdOn` | number | ✅ | 开灯阈值 lux（整数） |
| `thresholdOff` | number | ✅ | 关灯阈值 lux（整数） |
| `version` | number | ✅ | 配置版本号，单调递增，硬件用于判断是否更新 |
| `timestamp` | number | ✅ | Unix 毫秒时间戳 |

**硬件收到后：**
1. 比对 `version` 与本地存储的版本号，若 `version ≤ 本地版本` 则忽略（幂等）
2. 更新本地 mode 和阈值
3. 若 mode 从 `auto` 切为 `manual`：**立即停止**本地自动控制逻辑
4. 若 mode 从 `manual` 切为 `auto`：**立即启用**本地自动控制逻辑
5. 回复 `config/ack`（见 3.3）

---

### 3.2 配置请求 `devices/{id}/config/request`

```
硬件 → 后端
触发时机：硬件上电启动、MQTT 重连后
```

```json
{
  "deviceId": "lamp_001",
  "localVersion": 0,
  "timestamp": 1751436000000
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `deviceId` | string | ✅ | 设备 ID |
| `localVersion` | number | ✅ | 硬件当前持有的配置版本，`0` 表示无本地配置 |
| `timestamp` | number | ✅ | Unix 毫秒时间戳 |

**后端收到后：**
1. 从数据库查询该设备最新 mode + threshold
2. 若 `DB版本 > localVersion` 或 `localVersion === 0`：立即推送 `config`
3. 若 `DB版本 === localVersion`：无需推送（可选回复 `config` 做确认）

### 3.3 配置确认 `devices/{id}/config/ack`

```
硬件 → 后端
触发时机：硬件收到并应用 config 后
```

```json
{
  "deviceId": "lamp_001",
  "version": 3,
  "accepted": true,
  "timestamp": 1751436000000
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `deviceId` | string | ✅ | 设备 ID |
| `version` | number | ✅ | 确认的配置版本号 |
| `accepted` | boolean | ✅ | 是否接受配置 |
| `timestamp` | number | ✅ | Unix 毫秒时间戳 |

### 3.4 自动控制事件 `devices/{id}/auto-action`

```
硬件 → 后端  QOS 1
触发时机：硬件本地自动控制逻辑触发开关灯后
```

```json
{
  "deviceId": "lamp_001",
  "action": "on",
  "trigger": "light_below_threshold",
  "lightIntensity": 180,
  "thresholdOn": 200,
  "thresholdOff": 600,
  "timestamp": 1751436000000
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `deviceId` | string | ✅ | 设备 ID |
| `action` | string | ✅ | `"on"` 开灯 / `"off"` 关灯 |
| `trigger` | string | ✅ | `"light_below_threshold"` / `"light_above_threshold"` |
| `lightIntensity` | number | ✅ | 触发时的光照值 lux |
| `thresholdOn` | number | ✅ | 触发时本地存储的开灯阈值 |
| `thresholdOff` | number | ✅ | 触发时本地存储的关灯阈值 |
| `timestamp` | number | ✅ | Unix 毫秒时间戳 |

**后端收到后：**
1. 更新 `devices.current_state` 为 `action` 的值
2. 写入 `control_logs`（operator_id=0, operator_name='自动控制'）
3. 打印日志（见 5.5）

> **QOS 1 说明：** 自动控制事件可能因断网而丢失。服务端不依赖此事件来触发控制，仅用于事后记录。硬件不需要重发——下次上报 data/heartbeat 时后端可从中推断最新状态。

## 4. 硬件行为约束

### 4.1 必须实现的 MQTT Topic

| Topic | 方向 | QOS | 频率 | 说明 |
|-------|------|:---:|------|------|
| `devices/{id}/config` | 订阅 | 1 | 按需 | 接收后端配置下发 |
| `devices/{id}/config/request` | 发布 | 1 | 启动/重连时 | 请求最新配置 |
| `devices/{id}/config/ack` | 发布 | 1 | 收到 config 后 | 配置确认 |
| `devices/{id}/auto-action` | 发布 | 1 | 自动开关灯时 | 上报自动控制事件 |
| `devices/{id}/control` | 订阅 | 1 | 按需 | 接收手动控制指令 |
| `devices/{id}/control/response` | 发布 | 1 | 执行指令后 | 控制结果反馈 |
| `devices/{id}/data` | 发布 | 0 | 30~60s | 光照数据上报 |
| `devices/{id}/heartbeat` | 发布 | 0 | 30~60s | 心跳保活 |

### 4.2 上电/重连初始化流程

```
硬件启动
  │
  ├─ 1. 连接 MQTT Broker
  ├─ 2. 订阅 devices/{id}/config
  ├─ 3. 订阅 devices/{id}/control
  ├─ 4. 发布 devices/{id}/config/request（带上 localVersion）
  ├─ 5. 等待后端回复 devices/{id}/config
  │      └─ 超时 10 秒无回复 → 使用本地存储的旧配置（如有）
  ├─ 6. 启动光照传感器采集定时器
  └─ 7. 启动心跳定时器
```

### 4.3 自动控制决策循环（运行在硬件上）

```
每 30~60 秒：
  │
  ├─ 读取光照传感器 → lightIntensity
  ├─ 发布 devices/{id}/data（上报光照值）
  │
  └─ if mode === 'auto' && 本地已存储阈值:
        │
        ├─ if lightIntensity < thresholdOn && currentState === 'off':
        │     执行开灯
        │     currentState = 'on'
        │     发布 devices/{id}/auto-action { action: "on", ... }
        │
        ├─ if lightIntensity > thresholdOff && currentState === 'on':
        │     执行关灯
        │     currentState = 'off'
        │     发布 devices/{id}/auto-action { action: "off", ... }
        │
        └─ else: 不动作
```

### 4.4 防抖（运行在硬件上）

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `DEBOUNCE_COUNT` | 3 | 连续 N 次采样满足条件才触发 |

```
防抖逻辑（硬件本地维护一个滑动计数器）：
  consecutiveMatch = 0
  每次采样:
    if 满足开灯/关灯条件:
      consecutiveMatch++
    else:
      consecutiveMatch = 0
    if consecutiveMatch >= DEBOUNCE_COUNT:
      执行动作
      consecutiveMatch = 0
```

> 防抖由硬件本地实现，后端不参与判断。硬件可简化实现（如 `DEBOUNCE_COUNT=1` 即单次判断，立即触发）。

### 4.5 mode 切换时的行为

| 切换 | 硬件行为 |
|------|---------|
| `auto` → `manual` | 立即停止本地自动判断；保持当前开关状态不变；仅响应后端 control 指令 |
| `manual` → `auto` | 立即启用本地自动判断；从下一次光照采样开始执行阈值比对 |

### 4.6 设备 ID 命名

- **格式：** `lamp_` + 三位数字编号，如 `lamp_001`
- **原因：** 后端 MQTT 消息处理会**静默丢弃**非 `lamp_` 前缀的消息

### 4.7 光照传感器要求

| 指标 | 约束 |
|------|------|
| 上报/采样频率 | 30~60 秒 |
| 光照范围 | 0 ~ 65535 lux（整数） |
| 单位 | lux |
| data 消息字段名 | **`lightIntensity`**（优先）；兼容 `lux`、`value`、`light_intensity` |

### 4.8 指令执行要求

| 指标 | 约束 |
|------|------|
| 响应时间 | **< 3 秒**（> 10 秒后端判 timeout） |
| response 必须包含 | `requestId`（与指令一致）、`success` |
| 冲突处理 | 若硬件正在执行本地自动开灯，同时收到后端手动关灯指令 → **后端指令优先**，打断本地自动动作 |

## 5. 后端实现约束

### 5.1 配置同步：监听 mode/threshold API 变更

用户通过 REST API 修改模式或阈值时，后端 **在写库成功后立即推送 config 到硬件**：

```
POST /api/devices/{id}/threshold  → DB 写入 → MQTT publish devices/{id}/config
PUT  /api/devices/{id}/mode       → DB 写入 → MQTT publish devices/{id}/config
```

每次推送时 `version` 递增（可用 `thresholds.updated_at` 的 Unix 毫秒时间戳作为 version）。

### 5.2 配置同步：响应硬件 config/request

订阅 `devices/+/config/request`，收到后：

```
1. 提取 deviceId
2. 查询 devices.mode + thresholds 记录
3. 计算 version = max(thresholds.updated_at 的毫秒时间戳, 0)
4. if version > localVersion || localVersion === 0:
     publish devices/{id}/config (含 mode, thresholdOn, thresholdOff, version)
   else:
     无需推送（硬件已是最新）
```

### 5.3 接收 auto-action 事件

订阅 `devices/+/auto-action`，收到后：

```
1. 更新 devices.current_state = action
2. 写入 control_logs:
     device_id = deviceId
     command = action
     status = 'success'
     operator_id = 0
     operator_name = '自动控制'
     request_time = timestamp
     response_time = timestamp
     result_message = "光照 {lightIntensity} {高于/低于} 阈值 {thresholdOn/thresholdOff}"
3. 打印日志
```

### 5.4 移除旧的 TODO

`device-control.service.ts` 第 96 行的 TODO：
```typescript
// TODO: 自动控制 — 根据光照阈值和当前模式自动开关灯
```
应**删除**。`handleHardwareData()` 不再承担自动判断职责，保持只做数据入库。

### 5.5 日志规范

| 事件 | 级别 | 格式 |
|------|------|------|
| 推送配置 | info | `[ConfigSync] 📤 lamp_001 ← config v3 (auto, on=200, off=600)` |
| 收到配置请求 | info | `[ConfigSync] 📩 lamp_001 requested config (local v2, db v3) → pushing` |
| 收到 auto-action | info | `[AutoAction] 💡 lamp_001 自动开灯（光照 180 < 阈值 200）` |
| 收到 auto-action | info | `[AutoAction] 🌙 lamp_001 自动关灯（光照 650 > 阈值 600）` |
| 配置同步失败 | warn | `[ConfigSync] lamp_001 publish failed: MQTT not connected` |

## 8. 兼容性声明

- **REST API：零影响。** 11 个现有 API 接口不变。
- **数据库：零 DDL。** 不建新表、不改字段，所有自动控制数据写入现有表。
- **前端：零影响。** 控制日志中 `operator_name === '自动控制'` 即可区分来源。
- **MQTT：新增 3 个 topic**（config、config/request、auto-action），不影响现有 4 个 topic 的行为。
- **硬件现有命令响应协议：不受影响。** control → response 的格式和执行逻辑完全不变。
