### 数据底层说明
- 后端数据统一来自 MySQL 数据库 `dream23-28`（数据库服务器 `47.108.58.107:3306`）。
- 核心数据表包括：`users`、`devices`、`light_records`、`alerts`、`system_config`、`control_logs`。
- 实时光照数据目前由前端定时器每 5 秒调用 `GET /api/light/current` + `POST /api/light/record` 模拟产生。
- MQTT 仅后端用于下发控制指令，前端未直接订阅 MQTT。

## 三、还需要连接或完善的地方

### 3.1 前端有封装，但后端未实现

| 前端函数 | 接口 | 说明 |
|---|---|---|
| `resolveAlert(id, note)` | `PUT /api/alerts/:id/resolve` | 前端 API 文件已定义，但后端只有 `PUT /api/alerts/:id/handle`（不带备注参数）。如需“处理并填写备注”，需要后端补全该接口。 |

### 3.2 纯前端功能，无后端对接

| 功能 | 现状 | 建议 |
|---|---|---|
| **智能问答** | 使用前端本地硬编码 `ragKnowledge` 字典匹配 | 如需真正 RAG 知识库，需要新增后端接口（例如 `POST /api/chat`），将问题传给后端进行检索与回答。 |
| **顶部离线告警条** | 由 `devices` 列表中的 `online` 字段计算 | 若要与“告警日志”保持一致，应改为读取 `GET /api/alerts` 中未处理的“设备离线”告警。 |

### 3.3 实时性可以加强

- 当前光照“实时”数据依赖前端轮询 `GET /api/light/current` + `POST /api/light/record` 模拟。
- 建议：
  1. 让真实设备或模拟器定时调用 `POST /api/devices/:deviceId/heartbeat` 和 `POST /api/light/record` 上报数据；
  2. 或引入 MQTT over WebSocket / Socket.IO，由后端向前端推送实时光照、设备上下线、告警事件。

### 3.4 与 API 接口设计文档核对

- 建议对照 `任务3_API接口设计文档.md`，检查是否还有未落地的接口（例如：用户管理、角色权限细分、能耗统计、固件升级、批量阈值导入导出等）。
- 目前代码中已落地 24 个核心接口。
