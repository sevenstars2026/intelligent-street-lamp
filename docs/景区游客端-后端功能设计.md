# 景区游客端 — 后端功能设计

> 对应计划书 v1.1 | 设计日期：2026-07-08 | 最后更新：2026-07-09

---

## 一、设计原则

1. **遵循现有三层架构**：Routes → Controllers → Services，不引入新模式
2. **统一响应信封**：`{ code: number, message: string, data: any }`
3. ~~静态数据阶段不走 DB~~ → **已迁移到 DB**：景区数据在 `DatabaseService.init()` 中以 REPLACE INTO 种子写入，后续替换为真实数据时零代码改动
4. **复用已有基础设施**：告警写入复用 `alarms` 表，图片上传复用 Express 实例，认证复用 mock auth 中间件

### 4.3 故障上报接口

#### POST /api/reports（游客提交故障上报）

- **Content-Type**: `multipart/form-data`
- **字段**:

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| name | text | ✅ | 上报人姓名（2-20 字符） |
| phone | text | ✅ | 上报人手机号（11 位，`1\d{10}`） |
| lampId | text | ✅ | 故障路灯编号（格式 `lamp_XXX`） |
| description | text | ✅ | 故障描述（10-200 字） |
| photos | file[] | ❌ | 故障照片（最多 3 张，单张 ≤ 5MB） |

- **处理流程**:

```
1. multer 接收文件 → 保存到 uploads/（文件名: 时间戳-随机数.扩展名）
2. 校验必填字段 + 格式（name/phone/lampId/description）
3. ReportService.submit()
   ├── alarmId 暂为 0（未关联告警）
   ├── status 固定 'active'
   └── photoUrls 仅存文件名数组
4. 写入 fault_reports 表
5. 返回完整 FaultReport 对象
```

- **响应**:
```json
{
  "code": 200,
  "message": "上报成功，工作人员将尽快处理",
  "data": {
    "id": 1,
    "alarmId": 0,
    "reporterName": "张三",
    "reporterPhone": "13800138000",
    "lampId": "lamp_001",
    "description": "路灯闪烁严重",
    "photoUrls": ["1712345678901-123456.jpg"],
    "status": "active",
    "createdAt": "2026-07-09T18:30:00.000Z",
    "resolvedAt": null,
    "resolveNote": null
  }
}
```

> **与设计的差异**：表单字段名使用 `name`/`phone`/`photos`（设计为 `reporterName`/`reporterPhone`/`photo`）；照片存于 `uploads/` 而非 `uploads/reports/`；暂未同步写入 alarms 表（alarmId=0）。

#### GET /api/reports（管理员查询故障上报列表）🆕

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| status | string | ❌ | `active` / `resolved`（内存过滤） |
| lampId | string | ❌ | 按路灯编号筛选（SQL WHERE） |
| page | number | ❌ | 页码，默认 1 |
| pageSize | number | ❌ | 每页条数，默认 20，最大 100 |

响应格式：`{ code: 200, data: { reports: [...], pagination: {...} } }`

#### GET /api/reports/:id（查询单条上报详情）🆕

#### PUT /api/reports/:id/resolve（管理员处理上报）🆕

请求体：`{ note?: string }`

处理逻辑：
1. 先 `getFaultReport(id)` 检查是否存在 / 是否已处理
2. `UPDATE fault_reports SET status='resolved', resolved_at=NOW() WHERE id=? AND status='active'`
3. 并发安全：`AND status='active'` 防止重复处理

### 4.4 图片上传中间件配置（实际实施）

```typescript
// 在 report.routes.ts 中配置 multer
const upload = multer({
  storage: multer.diskStorage({
    destination: path.join(__dirname, '../../uploads'),   // 注意：直接存 uploads/ 根目录
    filename: (_req, file, cb) => {
      const unique = Date.now() + '-' + Math.round(Math.random() * 1e9);
      cb(null, unique + path.extname(file.originalname));
    },
  }),
  limits: { fileSize: 5 * 1024 * 1024, files: 3 },       // 单张 ≤5MB，最多 3 张
});

// 路由使用：upload.array('photos', 3)
```

> **与设计的差异**：存储路径简化为 `uploads/`（设计为 `uploads/reports/`）；未启用 `fileFilter` MIME 类型校验。

### 4.6 景区数据种子（DatabaseService.init()）

景区数据（路线/拍照点/活动/路灯坐标）通过 `REPLACE INTO` 写入 MySQL，每次启动覆盖旧数据：

- **emoji 替换**：MySQL utf8 下 emoji 可能导致 SQL 语法错误，已将 🍂→银杏、🏛→建筑、🏟→体育、🎵→音乐、🍜→美食、🌍→文化、🏃→运动
- **逐条 try-catch 容错**：每个 REPLACE 语句独立包裹，单表种子失败不阻断数据库连接
- **连接池 charset**：`utf8mb4`（`database.ts` 第 18 行），支持 4 字节 UTF-8 字符

## 六、告警系统集成

故障上报写入 `alarms` 表时，使用以下字段映射：

| alarms 表字段 | 值 |
|---|---|
| deviceId | `report.lampId` |
| alarmType | `'fault_report'` |
| alarmLevel | `'medium'` |
| status | `'active'` |
| message | `"游客 {name}({phone}) 上报故障: {description}"` |
| createdAt | 当前时间 |

管理端 `AlarmPage.vue` 无需任何改动，因为告警列表已支持展示全部 alarmType，`fault_report` 类型的告警会直接出现在列表中。
