# 智慧路灯故障上报 AI 合规校验（DeepSeek API 三阶段审核）

## 整体架构流程

游客端提交故障上报接口 `POST /api/reports` → Express 后端统一拦截 → 代码规则校验 → DeepSeek AI 语义审核（OpenAI 兼容接口直连）→ 三态分流 → 人工审核 → 入库 + 生成告警：
1. **AI 驳回**（`auditPass=0`）：直接返回错误，**不入库故障上报、不生成告警、不保留图片**
2. **AI 通过**（`auditPass=1`）：进入人工审核队列（`reviewStatus=pending_review`），写入审计日志
3. **AI 不确定**（`auditPass=2`）：AI 异常/超时时兜底，标记为待人工审核

### 1. 环境配置新增（必做）

### 2. 新增 AI 审核 HTTP 调用模块（必做）
文件：`task3-backend/src/utils/http-client.ts`
核心功能：
- 调用 DeepSeek `/v1/chat/completions`（OpenAI 兼容格式）发起审核
- 使用 `AbortController` 实现超时控制（默认 15s）
- 构造 System Prompt（含 5 条审核规则） + User Message（上报内容）发送给 AI
- 解析 AI 返回的 JSON（兼容 markdown 代码块包裹），提取 `{ pass: boolean, reason: string }`
- 统一返回 `AuditResult` 三态结构：`{ pass: true | false | null, reason, rawResponse }`
- 异常兜底：API 不可用/超时/格式异常时返回 `pass: null`，标记人工复核

```typescript
// AuditResult 三态结构
export interface AuditResult {
  pass: boolean | null;  // true=通过, false=驳回, null=AI不确定(异常/超时)
  reason: string;
  rawResponse: string | null;
}
```

### 3. 新建数据库校验日志表 report_audit_log（必做）

  audit_pass TINYINT NOT NULL COMMENT 'AI校验结果: 0=驳回, 1=通过, 2=AI不确定(转人工)',
  audit_reason TEXT NOT NULL COMMENT 'AI校验结果说明',
  ai_response TEXT COMMENT 'AI审核原始返回',
  review_status VARCHAR(20) NOT NULL DEFAULT 'pending_review' COMMENT '人工审核状态',
  reviewer_id INT NULL COMMENT '审核人ID',
  reviewer VARCHAR(50) NULL COMMENT '审核人姓名',
  review_time DATETIME NULL COMMENT '审核时间',

### 4. 改造故障上报业务层（核心必改）
文件：`src/controllers/report.controller.ts`、`src/services/report.service.ts`
原有 `createReport` 流程重构，执行顺序：
1. multer 接收表单：姓名、手机号、路灯编号、故障描述、图片文件
2. **代码规则校验**：姓名 2-20 字符、手机号 11 位格式、路灯编号 `lamp_XXX` 格式、故障描述 10-200 字
3. **路灯存在性校验**：查询数据库确认 lampId 真实存在
4. **24h 重复检测**：同一手机号+路灯+描述 24h 内不重复提交
5. 调用 `callDeepSeekAudit()` 发起 AI 语义审核
6. 捕获 AI 调用异常（超时/服务不可用），三态分流：
   - `auditPass=0`（驳回）：写入日志 → 删除已上传图片 → 返回 `code:4001`，终止流程
   - `auditPass=1`（通过）或 `auditPass=2`（AI 不确定）：写入日志 → 保留图片 → 返回成功，进入人工审核队列
7. 调用 `ReportAuditService.createAuditLog()` 写入本次完整校验日志

## 二、DeepSeek AI 审核 Prompt 设计

### System Prompt（审核规则）：
```
你是一个智慧路灯系统的故障上报审核助手。请审核以下游客提交的故障上报内容是否合规。

审核规则：
1. 必须是真实的路灯故障描述（如：不亮、闪烁、灯杆倾斜、灯罩破损、线路外露等）
2. 不得包含广告、推销、联系方式
3. 不得包含政治敏感、色情、暴力内容
4. 不得是无意义的乱码或灌水文字
5. 如果是描述路灯故障，即使表达不够专业（如"灯坏了""不亮了"），也应判定通过

请严格以 JSON 格式回复，不要包含其他文字：
{ "pass": true/false, "reason": "审核说明（不通过时说明具体原因）" }
```

### API 调用参数：
```
model: deepseek-chat
temperature: 0.1   （低温度确保输出稳定一致）
max_tokens: 256    （仅需返回简短 JSON）
```

## 三、完整数据流（改造后）
1. 游客前端发起 `POST /api/reports` 携带表单+图片
2. Express 后端 multer 解析文件
3. 代码层规则校验（字段格式、路灯存在性、24h 重复检测）
4. ReportController 调用 `callDeepSeekAudit()` 直连 DeepSeek `/v1/chat/completions`
5. 获取 AI 校验结果，调用 `ReportAuditService.createAuditLog()` 写入 `report_audit_log` 表
6. 三态分支逻辑：
   - **AI 驳回**（`auditPass=0`, `reviewStatus=ai_rejected`）：删除已上传图片 → 返回 `{code:4001, message:驳回原因}`，流程结束
   - **AI 通过**（`auditPass=1`, `reviewStatus=pending_review`）：保留图片 → 返回 `{code:200, data:{auditId, reviewStatus}}`，等待人工审核
   - **AI 不确定**（`auditPass=2`, `reviewStatus=pending_review`）：同通过逻辑，人工审核时需特别关注
7. 管理端通过审核接口查询 `report_audit_log`，进行人工审核（通过/驳回），审核通过后回填 `fault_report_id`、`alarm_id`
8. 审核通过后同步写入 `fault_reports` 表 + `alarms` 告警表
