# 智慧路灯故障上报 AI 合规校验完整实现路线（DeepSeek API 三阶段审核）

## 整体架构流程

游客端提交故障上报接口 `POST /api/reports` → Express 后端统一拦截 → 代码规则校验 → DeepSeek AI 语义审核 → 三态分流 → 人工审核 → 入库 + 生成告警：
1. 校验不通过：直接返回错误，**不入库故障上报、不生成告警**
2. 校验通过：执行原有业务逻辑（存图片、写入fault_reports、同步生成alarms告警）

## 一、后端新增/修改内容清单（分必做/可选）
### 1. 环境配置新增（必做）
文件：`task3-backend/.env`
新增MaxKB工作流调用参数：
```env
# MaxKB AI合规校验配置

MAXKB_WORKFLOW_AUDIT_ID=你的工作流ID
MAXKB_WORKFLOW_TOKEN=工作流API密钥
# AI校验超时时间
MAXKB_TIMEOUT=10000
```

### 2. 新增通用HTTP请求工具（必做）
新建文件：`task3-backend/src/utils/http-client.ts`
封装Axios实例，统一请求MaxKB，处理超时、返回格式化、异常捕获，供上报服务调用。

### 3. 新建数据库校验日志表 report_audit_log（必做）
#### 表结构（MySQL）
```sql
CREATE TABLE `report_audit_log` (
  id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '日志主键',
  report_name VARCHAR(50) NOT NULL COMMENT '上报人姓名',
  report_phone VARCHAR(20) NOT NULL COMMENT '上报手机号',
  fault_content TEXT NOT NULL COMMENT '原始故障描述',
  audit_pass TINYINT NOT NULL COMMENT '校验是否通过 0驳回 1通过',
  audit_reason TEXT NOT NULL COMMENT 'AI校验结果说明',
  maxkb_response TEXT COMMENT 'MaxKB原始返回JSON',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '校验时间',
  INDEX idx_phone (report_phone),
  INDEX idx_create_time (create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT '故障上报AI合规校验日志';
```
#### 配套后端改动
1. `src/types/database.types.ts`：新增ReportAuditLog类型定义
2. `src/config/database.ts`：复用现有连接池，新增日志CRUD方法
3. 新建 `src/services/report-audit.service.ts`：封装日志新增查询方法，解耦上报主逻辑

### 4. 改造故障上报业务层（核心必改）
文件：`src/services/report.ts`、`src/controllers/report.ts`
原有 `createReport` 流程重构，执行顺序：
1. multer接收表单：姓名、手机号、故障描述、图片文件
2. 组装校验入参，调用 `http-client` 请求MaxKB工作流API
3. 捕获MaxKB调用异常（超时/服务不可用），自定义兜底策略（如标记人工复核）
4. 调用 `report-audit.service` 写入本次完整校验日志
5. 分支判断：
   - 分支A（不通过）：直接返回统一错误 `code:400`，携带驳回reason，终止流程
   - 分支B（通过）：执行原有逻辑：保存图片、插入fault_reports、生成alarms告警、返回成功数据

### 5. 统一错误码扩展（优化，建议做）
在后端统一枚举文件新增校验失败错误码，方便前后端识别：
```typescript
enum ErrorCode {
  AUDIT_REJECT = 4001, // 上报内容合规校验驳回
  MAXKB_SERVICE_ERROR = 5001 // AI校验服务异常
}
```

## 二、MaxKB工作流配置（独立操作，不改动项目代码）
1. 新建工作流，触发方式选择 **Web API**
2. 定义入参字段：`report_name`、`report_phone`、`fault_content`
3. 拖拽编排校验节点（顺序固定）：
   ① 基础字段非空校验
   ② 自定义敏感词库拦截节点
   ③ LLM文本审核节点，绑定《故障上报合规规范》知识库，语义校验违规内容
4. 输出固定结构化JSON：
```json
{
  "pass": true/false,
  "reason": "校验说明，如：包含违规描述/内容符合上报规范"
}
```
5. 发布工作流，复制调用地址、Token填入后端.env配置

## 三、完整数据流（改造后）
1. 游客前端发起 `POST /api/reports` 携带表单+图片
2. Express后端中间件鉴权、multer解析文件
3. ReportService调用http-client请求MaxKB工作流接口
4. 获取AI校验结果，调用ReportAuditService写入report_audit_log表
5. 分支逻辑：
   - 校验驳回：返回 `{code:4001, message:"上报内容不合规", data:{reason:xxx}}`，流程结束
   - 校验通过：执行原有图片存储 → 插入fault_reports → 同步写入alarms告警表 → 返回成功
6. 管理端可新增【AI校验日志查询页面】（可选扩展），通过接口查询report_audit_log记录，追溯驳回上报记录
