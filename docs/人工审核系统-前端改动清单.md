### 功能要求

#### 顶部工具栏
- **左侧**：标题"上报审核" + 待审核数量 badge
- **右侧**：
  - AI 校验结果筛选下拉框：全部 / ✅AI通过 / ⚠️AI不确定 / ❌AI驳回
  - 审核状态筛选下拉框：全部 / 待审核 / 已通过 / 已驳回
  - 刷新按钮

#### 表格列

| 列 | 内容 | 说明 |
|----|------|------|
| ID | `log.id` | 等宽字体 |
| 上报人 | `log.reportName` | |
| 手机号 | 脱敏显示 `138****5678` | 前端 maskPhone 处理 |
| 故障路灯 | `log.lampId`（可从 faultContent 中提取或新增字段） | 等宽字体 |

> **注意**：后端 `report_audit_log` 表未存储 `lampId` 字段（信息包含在 `fault_content` 中）。如果前端审核列表需要显示路灯编号列，有两种方案：
> - **方案 1（推荐）**：在后端 `report_audit_log` 表新增 `lamp_id VARCHAR(64)` 字段，上报时从请求体提取写入
> - **方案 2**：不在表格列中展示路灯编号，点击详情弹窗时展示完整描述（含路灯编号）

| AI 结果 | 图标 + 文字：✅AI通过 / ⚠️AI不确定 / ❌AI驳回 | 根据 auditPass 字段 |
| 故障描述 | `log.faultContent`，截断 30 字 | title 悬浮显示全文 |
| 上报时间 | `log.createTime`，格式化为 `YYYY-MM-DD HH:mm` | |
| 审核状态 | 标签：待审核(橙) / 已通过(绿) / 已驳回(红) / AI驳回(灰) | |
| 操作 | "审核"按钮（仅 `pending_review` 状态显示） | 点击打开详情弹窗 |

弹窗功能：
- 展示上报人信息 + 完整故障描述
- 展示 AI 校验结果（通过/不确定/驳回 + AI 给出的 reason）
- 审核意见文本框（选择"驳回"时必填校验，未填则弹出提示）
- 审核通过 → 调用 `PUT /api/review/:id/approve`，成功后刷新列表 + toast 提示
- 审核驳回 → 调用 `PUT /api/review/:id/reject`，携带 `reviewReason`，成功后刷新列表 + toast 提示
- 已审核的记录（approved/rejected/ai_rejected）也可以点击查看，但不显示操作按钮，改为显示审核历史：
  ```

  ```

## 四、修改 `src/utils/api.ts`

新增 4 个审核 API 函数：

```typescript
// ===== 人工审核（4 个） =====

// 获取审核列表
export function getReviewList(params?: {
  reviewStatus?: string;
  auditPass?: number;
  page?: number;
  pageSize?: number;
}) {
  return api.get('/review/list', { params });
}

// 获取审核详情
export function getReviewDetail(id: number) {
  return api.get(`/review/${id}`);
}

// 审核通过
export function approveReview(id: number) {
  return api.put(`/review/${id}/approve`);
}

// 审核驳回
export function rejectReview(id: number, reviewReason: string) {
  return api.put(`/review/${id}/reject`, { reviewReason });
}
```

响应拦截器已处理统一信封格式 `{ code, message, data }`，无需额外处理。

## 七、游客端改动

**无**。游客提交上报的交互完全不变。AI 校验对游客透明：
- AI 直接驳回 → 游客看到"内容不合规，请修改后重新提交"（前端拦截 `code: 4001`）
- 通过/不确定 → 游客看到"上报已提交，工作人员将尽快处理"（与现有文案一致）

`tourist-frontend/src/utils/api.ts` 无需修改（`Content-Type` 修复已在当前版本完成）。
