# 人工审核系统 — 后端改动清单

> **前置依赖**：本文档基于 `docs/智能检测系统计划书.md` 的 AI 校验方案，在其基础上叠加人工审核。
>
> **核心流程**：游客提交上报 → AI 合规校验 → 三态分流 → 人工审核 → 入库 + 生成告警

---

## 完整数据流（改造后）

```
游客 POST /api/reports (multipart)
      │
      ▼
  multer 解析文件
      │
      ▼
  AI 合规校验（调用 MaxKB 工作流 API）
      │
      ├─ AI 返回 pass=false ──→ 写入 report_audit_log (audit_pass=0)
      │                         返回 { code: 4001, reason: "..." }
      │                         ⛔ 不入库 fault_reports，不生成告警
      │
      ├─ AI 返回 pass=true  ──→ 写入 report_audit_log (audit_pass=1)
      │                         设置 review_status = 'pending_review'
      │                         ⛔ 暂不入库 fault_reports，等待人工审核
      │
      └─ AI 超时/异常 ──────→ 写入 report_audit_log (audit_pass=2 表示不确定)
                              设置 review_status = 'pending_review'
                              ⛔ 暂不入库 fault_reports，等待人工审核
      
      ▼
  市政人员在管理端审核页面操作：

  ├─ 审核通过 ──→ report_audit_log: review_action='approved'
  │               ↓
  │          写入 fault_reports (alarm_id 指向新告警)
  │               ↓
  │          写入 alarms (alarm_type='fault_report', level='medium')
  │               ↓
  │          返回成功
  │
  └─ 审核驳回 ──→ report_audit_log: review_action='rejected' + review_reason
                  ⛔ 不入库 fault_reports，不生成告警
                  驳回原因仅市政人员可见
```

## 二、新增文件清单（4 个新文件）

### 2.1 `task3-backend/src/utils/http-client.ts`

**用途**：通用 HTTP 请求工具，封装 Axios 实例，专门用于调用 MaxKB 工作流 API。

**功能要求**：
- 创建 Axios 实例，baseURL 读取 `MAXKB_BASE_URL`
- 默认超时时间读取 `MAXKB_TIMEOUT`（默认 10000ms）
- 封装 `callMaxKBWorkflow(workflowId, token, params)` 方法
- POST 请求到 `/api/v1/workflow/{workflowId}/run`，携带 Authorization Bearer token
- 请求体格式：`{ inputs: { report_name, report_phone, fault_content } }`
- 响应体解析：提取 `data.output.pass` 和 `data.output.reason`
- 异常处理：超时、网络错误、返回格式异常 → 统一抛出带错误码的异常
- 超时和网络异常时返回 `{ pass: null, reason: 'AI校验服务不可用' }`（触发人工审核兜底）

### 2.2 `task3-backend/src/services/report-audit.service.ts`

**用途**：封装 `report_audit_log` 表的增删改查操作，解耦上报主逻辑。

**需要实现的方法**：

```typescript
class ReportAuditService {
  // 写入一条 AI 校验日志。返回插入后的完整记录（含自增 ID）
  static async createAuditLog(data: {
    reportName: string;
    reportPhone: string;
    faultContent: string;
    auditPass: 0 | 1 | 2;      // 0=驳回, 1=通过, 2=不确定(AI异常/超时)
    auditReason: string;
    maxkbResponse?: string;     // MaxKB 原始返回 JSON
    reviewStatus: 'ai_rejected' | 'pending_review';
  }): Promise<ReportAuditLog>;

  // 分页查询审核日志（供管理端审核页面使用）
  static async listAuditLogs(filters: {
    reviewStatus?: string;       // 'pending_review' | 'approved' | 'rejected' | 'ai_rejected'
    auditPass?: number;          // 0 | 1 | 2 — AI 校验结果筛选
    page?: number;
    pageSize?: number;
  }): Promise<{ logs: ReportAuditLog[]; pagination: {...} }>;

  // 查询单条审核日志
  static async getAuditLogById(id: number): Promise<ReportAuditLog | null>;

  // 人工审核（通过/驳回）
  static async reviewAuditLog(
    id: number,
    action: 'approved' | 'rejected',
    reviewer: string,           // 审核人姓名（从当前登录用户取）

  ): Promise<ReportAuditLog>;

  // 检查同一手机号+同一路灯+同一描述的重复上报（24小时内），返回最近一条

    phone: string,
    lampId: string,
    description: string
  ): Promise<ReportAuditLog | null>;
}
```

### 2.3 `task3-backend/src/controllers/review.controller.ts`

**用途**：人工审核相关的 HTTP 接口处理。

**需要实现的方法**：

```typescript
class ReviewController {
  // GET /api/review/list — 审核列表（支持 reviewStatus、auditPass、page、pageSize 筛选）
  static async listReview(req, res);

  // GET /api/review/:id — 查看单条审核详情（含上报信息、照片、AI 校验结果）
  static async getReview(req, res);

  // PUT /api/review/:id/approve — 审核通过
  //   1. 调用 ReportAuditService.reviewAuditLog(id, 'approved', reviewer)
  //   2. 调用 ReportService 写入 fault_reports（状态 active）
  //   3. 调用 AlarmService 写入 alarms（alarm_type='fault_report', level='medium'）
  //   4. 将 alarm.id 回写到 fault_reports.alarm_id
  //   5. 注意：整个操作需要事务保证（审核日志更新+上报入库+告警生成 全成功或全回滚）
  static async approveReview(req, res);

  // PUT /api/review/:id/reject — 审核驳回
  //   1. 校验 reviewReason 必填
  //   2. 调用 ReportAuditService.reviewAuditLog(id, 'rejected', reviewer, reason)
  //   3. 不回写 fault_reports 和 alarms
  static async rejectReview(req, res);

  // POST /api/review/batch — 批量审核（可选，人力充裕时做）
  static async batchReview(req, res);
}
```

### 3.4 `task3-backend/src/controllers/report.controller.ts`

**改造 `submitReport` 方法**：

原有逻辑（校验 → 入库）改为：
1. multer 接收表单（不变）
2. 参数校验（不变）
3. **新增：查重检查**（调用 `ReportAuditService.findDuplicateReport`，24h 内相同手机+路灯+描述则拒绝）
4. **新增：调用 MaxKB AI 校验**（`http-client` → MaxKB 工作流）
5. **新增：写入 `report_audit_log`**（audit_pass + maxkb_response）

   - `audit_pass === 0`（AI 驳回）→ 返回 `{ code: 4001, message: auditReason }`
   - `audit_pass === 1 或 2`（通过/不确定）→ 返回 `{ code: 200, message: '上报已提交，等待审核' }`
   - **不再直接调用 `ReportService.submit`**（改由审核通过后调用）

## 四、错误码扩展

在现有错误响应格式基础上，新增：

| 错误码 | 含义 | 使用场景 |
|--------|------|---------|
| `4001` | AI 合规校验驳回 | AI 判定内容不合规，直接拒绝 |
| `4002` | 重复上报 | 24h 内相同内容已提交过 |
| `5001` | AI 校验服务异常 | MaxKB 不可用，已自动转为人工审核（对游客透明） |

## 五、事务要求

审核通过接口 `PUT /api/review/:id/approve` 需要保证原子性：

```
BEGIN TRANSACTION
  1. UPDATE report_audit_log SET review_status='approved', review_action='approved', ...
  2. INSERT INTO alarms (...alarm_type='fault_report'...)
  3. INSERT INTO fault_reports (...alarm_id=上一步的id...)
COMMIT
```

任一步失败则全部回滚。使用 `mysql2/promise` 的 `pool.getConnection()` + `conn.beginTransaction()` + `conn.commit()` / `conn.rollback()` 实现。
