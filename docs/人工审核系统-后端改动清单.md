# 人工审核系统 - 后端改动清单（修订版）

> 修订日期：2026-07-11
>
> 适用目录：`task3-backend`
>
> 实现基线：当前项目使用 Express、TypeScript、`mysql2/promise`、Node.js 20，数据库不可用时会回退到 `MockDatabase`。项目当前只有 Mock JWT，没有真实认证中间件。

## 一、目标与范围

将游客故障上报从“校验后直接写入 `fault_reports`”改为两阶段流程：

1. 游客提交姓名、手机号、路灯编号、故障描述和照片。
2. 系统检查 24 小时内的重复上报。
3. 系统调用 MaxKB 工作流进行 AI 合规校验。
4. AI 明确驳回时记录审核日志并拒绝本次上报。
5. AI 通过、超时、网络异常或结果无法解析时，保存完整上报数据并进入人工待审核状态。
6. 市政人员审核通过后，在同一数据库事务中创建告警和故障上报，并更新审核状态。
7. 市政人员驳回后只更新审核记录，不创建告警和故障上报。

批量审核不在本次范围内。真实 JWT 接入不在本次范围内，但审核路由必须使用独立角色校验中间件；当前 Mock JWT 下允许 `admin` 和 `municipal`。

## 二、确定的数据流

```text
POST /api/reports (multipart)
  -> multer 保存最多 3 张照片
  -> 参数校验
  -> 24 小时重复检查（手机号 + 路灯编号 + 规范化描述）
     -> 重复：删除本次新上传文件，返回业务码 4002
  -> 调用 MaxKB
     -> pass=false：写入 ai_rejected 审核日志，删除本次上传文件，返回业务码 4001
     -> pass=true：写入 pending_review 审核日志，保留照片，返回 code=200
     -> 超时/网络错误/结果异常：以 audit_pass=2 写入 pending_review，保留照片，返回 code=200

PUT /api/review/:id/approve
  -> 校验角色 admin/municipal
  -> 开启事务并锁定审核记录
  -> 仅允许 pending_review -> approved
  -> 查询路灯名称
  -> INSERT alarms，取得 alarm_id
  -> INSERT fault_reports，引用 alarm_id
  -> UPDATE report_audit_log，写 reviewer/review_time/review_action
  -> COMMIT

PUT /api/review/:id/reject
  -> 校验角色 admin/municipal 和非空 reviewReason
  -> 条件更新 pending_review -> rejected
  -> 保留审核日志；删除其待审核照片
```

### 4.2 新增数据库表

在 `DatabaseService.init()` 中、`fault_reports` 建表之后创建：

```sql
CREATE TABLE IF NOT EXISTS report_audit_log (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  report_name VARCHAR(100) NOT NULL,
  report_phone VARCHAR(30) NOT NULL,
  lamp_id VARCHAR(64) NOT NULL,
  fault_content TEXT NOT NULL,
  photo_urls JSON NULL,
  audit_pass TINYINT NOT NULL,
  audit_reason TEXT NOT NULL,
  maxkb_response LONGTEXT NULL,
  review_status ENUM('ai_rejected','pending_review','approved','rejected') NOT NULL,
  reviewer_id INT NULL,
  reviewer VARCHAR(100) NULL,
  review_time DATETIME NULL,
  review_action ENUM('approved','rejected') NULL,
  review_reason TEXT NULL,
  fault_report_id INT NULL,
  alarm_id INT NULL,
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_audit_status_time (review_status, create_time),
  INDEX idx_audit_pass (audit_pass),
  INDEX idx_audit_duplicate (report_phone, lamp_id, create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

约束：

- `audit_pass` 只能由服务层写入 0、1、2。
- `photo_urls` 保存 multer 生成的文件名数组，使审核详情和审核通过后的故障上报可以复用照片。
- `fault_report_id` 和 `alarm_id` 用于追踪审批结果和保证接口幂等。
- 不删除或重建已有业务表。本次仅使用 `CREATE TABLE IF NOT EXISTS` 创建新表。

## 五、新增模块

### 5.1 `src/utils/http-client.ts`

使用 Node.js 20 原生 `fetch` 和 `AbortController`，不新增 Axios 依赖。

导出：

```typescript
interface AuditWorkflowResult {
  pass: boolean | null;
  reason: string;
  rawResponse: string | null;
}

async function callMaxKBWorkflow(params: {
  reportName: string;
  reportPhone: string;
  lampId: string;
  faultContent: string;
}): Promise<AuditWorkflowResult>;
```

请求要求：

- `POST {MAXKB_BASE_URL}/api/v1/workflow/{MAXKB_WORKFLOW_AUDIT_ID}/run`
- Header：`Authorization: Bearer {MAXKB_WORKFLOW_TOKEN}`、`Content-Type: application/json`
- Body：`{ "inputs": { "report_name", "report_phone", "lamp_id", "fault_content" } }`
- 兼容读取 `data.output` 为对象或 JSON 字符串。
- 只有明确的布尔值 `true`/`false` 才作为确定结果；缺字段、非 2xx、超时和网络错误统一返回 `pass: null`，不能抛出到游客接口造成 500。
- `rawResponse` 应限制长度，避免异常响应无限写入数据库；不得记录 Token。
- 缺少 URL、工作流 ID 或 Token 时视为服务不可用，进入人工审核兜底。

### 5.2 `src/services/report-audit.service.ts`

实现：

```typescript
createAuditLog(data): Promise<ReportAuditLog>
listAuditLogs(filters, page, pageSize): Promise<{ logs: ReportAuditLog[]; pagination: Pagination }>
getAuditLogById(id): Promise<ReportAuditLog | null>
findDuplicateReport(phone, lampId, description, since): Promise<ReportAuditLog | null>
approveAuditLog(id, reviewer): Promise<{ auditLog; alarm; report }>
rejectAuditLog(id, reviewer, reviewReason): Promise<ReportAuditLog>
```

规则：

- 查重范围包含 `pending_review`、`approved` 和 `ai_rejected`，不包含人工 `rejected`。
- 描述比较使用去除首尾空白后的完整字符串；暂不做模糊匹配。
- 分页参数：`page >= 1`，`1 <= pageSize <= 100`；数据库执行 `COUNT(*)` 和 `LIMIT/OFFSET`，不将全表读入内存。
- 审核通过只允许处理 `pending_review`。不存在返回 404，已处理返回 409。
- 驳回原因去除首尾空白后必须为 1-500 字符。

{ "code": 200, "message": "success", "data": {} }

参数错误使用 HTTP 400，不存在使用 HTTP 404，重复审核使用 HTTP 409，服务器错误使用 HTTP 500。

## 六、现有模块修改

### 6.1 `src/services/database.service.ts`

- 增加审核日志行映射、创建、详情、筛选分页、计数和查重方法。
- 增加 `approveReportAudit()`：内部使用 `pool.getConnection()`、`beginTransaction()`、`SELECT ... FOR UPDATE`、`commit()` 和 `rollback()`。
- 审核通过事务固定顺序：锁审核记录 -> 插入告警 -> 插入故障上报 -> 更新审核日志。
- 所有事务 SQL 必须使用同一个 `PoolConnection`，不得在事务中调用使用连接池的普通 CRUD 方法。
- 更新审核状态时同时限制 `review_status='pending_review'`，避免并发重复审批。
- 数据库不可用时，`MockDatabase` 提供语义一致的审核操作；内存操作同步完成，失败时不得留下部分数据。

### 6.2 `src/mock/mock-database.ts`

- 增加审核日志集合、自增 ID、筛选、查重、审核通过和驳回。
- `reset()` 必须清空审核日志并重置 ID。
- Mock 审核通过也要创建 `fault_report` 和 `alarm`，并回写两个 ID。

### 6.3 `src/services/report.service.ts`

- 保留现有 `submit()` 供兼容测试，但游客控制器不再调用它。
- 新增或改造为可复用的审核后创建数据构造逻辑；真实事务内的 SQL 由 `DatabaseService` 执行。
- 禁止捕获真实数据库写入错误后静默写入 Mock，避免同一次业务请求跨存储。存储选择只由 `DatabaseService.init()` 决定。

### 6.4 `src/controllers/report.controller.ts`

- 保留现有参数校验并对字符串先做类型检查，避免对非字符串调用 `.length`。

- AI 通过或兜底：HTTP 200，提示“上报已提交，等待审核”，返回审核记录 ID 和状态，不返回 AI 原始响应。
- 参数错误、重复或 AI 驳回时删除本次上传的照片；写日志失败时也删除，避免孤立文件。

## 七、错误与响应语义

| HTTP | 业务码 | 场景 |
|---|---:|---|
| 200 | 200 | 提交进入待审核、查询成功、审核成功 |
| 400 | 400 | 参数格式错误、驳回原因缺失 |
| 400 | 4001 | AI 明确判定不合规 |
| 409 | 4002 | 24 小时内重复上报 |
| 401 | 401 | 未认证 |
| 403 | 403 | 角色无审核权限 |
| 404 | 404 | 审核记录不存在 |
| 409 | 409 | 审核记录已处理 |
| 500 | 500 | 数据库或未预期的服务器错误 |

AI 服务异常不向游客返回 5001。它记录为 `audit_pass=2`，对游客返回正常的“等待审核”。如需运维监控，通过服务日志区分超时、网络错误、HTTP 错误和解析错误。

## 九、测试与验收标准

### 9.1 自动化测试

必须覆盖：

1. MaxKB 返回明确通过、明确驳回、JSON 字符串 output、超时、非 2xx 和畸形响应。
2. 重复检查命中与超过 24 小时不命中。
3. AI 驳回写入 `ai_rejected`，且不创建故障上报或告警。
4. AI 通过和 AI 异常均进入 `pending_review`，照片字段完整。
5. 列表按状态和 AI 结果筛选，分页总数正确。
6. 审核详情不存在时返回 404。
7. 审核通过创建一个告警和一个故障上报，并正确互相关联。
8. 事务中任一步失败时审核状态、告警和故障上报全部回滚。
9. 同一记录重复或并发审核时只允许一次成功，后续返回 409。
10. 驳回原因校验、驳回后不生成业务数据。
11. admin/municipal 可访问，其他角色 403，无用户 401。
12. TypeScript 编译通过，现有接口回归测试通过。

### 9.2 验收命令

```bash
cd task3-backend
npm run build
npm test -- --runInBand
```

在具备测试数据库时再执行集成验证：初始化新表、提交待审核记录、审核通过、核对三张表关联、模拟事务失败并确认回滚。没有可用 MaxKB 时使用本地 HTTP mock 验证所有 AI 分支；真实 MaxKB 联调作为环境验收项，不阻塞离线自动化测试。

## 十一、实施前提与剩余外部依赖

- MaxKB 实例的真实 URL、工作流 ID、Token 和实际响应样例由部署环境提供。
- 当前文档假设工作流路径为 `/api/v1/workflow/{id}/run`。实现时必须通过可配置客户端和 mock 测试隔离该假设；真实联调前需用部署实例确认路径和鉴权格式。

- MySQL 集成测试需要独立测试库，不能对生产库执行失败注入或清理测试数据。
