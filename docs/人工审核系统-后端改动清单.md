# 人工审核系统 — 后端改动清单

> **核心流程**：游客提交上报 → 代码规则校验 → DeepSeek AI 语义审核 → 三态分流 → 人工审核 → 入库 + 生成告警

---

## 完整数据流（改造后）

```
游客 POST /api/reports (multipart)
      │
      ▼
  multer 解析文件
      │
      ▼
  代码规则校验（ComplianceCheckService.checkBasic）
      │
      ├─ 敏感词/垃圾内容/重复上报/频率超限
      │    → 写入 report_audit_log (audit_pass=0)
      │    → 返回 { code: 4001, reason: "..." }
      │
      ▼ 规则通过
  DeepSeek AI 语义审核（ComplianceCheckService.checkAI）
      │  调用 DeepSeek chat/completions API
      │  提示词：判断故障描述是否真实、合规
      │
      ├─ AI 判定违规 → 写入 report_audit_log (audit_pass=0)
      │                返回 { code: 4001, reason: "AI审核未通过" }
      │
      ├─ AI 判定通过 → 写入 report_audit_log (audit_pass=1)
      │                review_status = 'pending_review'
      │                返回 "上报已提交，等待审核"
      │
      └─ AI API 超时/异常 → 写入 report_audit_log (audit_pass=2)
                            review_status = 'pending_review'
                            返回 "上报已提交，等待审核"（透明降级）

      ▼
  市政人员在管理端审核页面操作：

  ├─ 审核通过 → report_audit_log: review_action='approved'
  │              写入 fault_reports (alarm_id 指向新告警)
  │              写入 alarms (alarm_type='fault_report', level='medium')
  │
  └─ 审核驳回 → report_audit_log: review_action='rejected' + review_reason
                 不入库 fault_reports，不生成告警
                 驳回原因仅市政人员可见
```

### 2.1 `task3-backend/src/services/compliance-check.service.ts`

**用途**：故障上报合规校验引擎。封装代码规则校验 + DeepSeek AI 语义审核。

**实现要点**：

```typescript
export class ComplianceCheckService {
  // ========== 第一层：代码规则校验 ==========

  // 敏感词库（按需扩充）
  private static SENSITIVE_WORDS: string[] = [
    '加微信', '加V', '扫码', '免费领', '点击链接', '公众号',
    '办证', '贷款', '兼职', '刷单', '代开发票',
  ];

  // 垃圾内容正则
  private static SPAM_PATTERNS: RegExp[] = [
    /^(\S)\1{10,}$/,              // 单字符重复 ≥10 次
    /(.)\1{6,}/,                   // 任意字符连续重复 ≥6 次
    /^[a-zA-Z\s]{10,}$/,           // 纯英文内容
    /^[\d\s.,，。、；;：:！!？?\-—]+$/,  // 纯数字+标点
  ];

  // 规则校验（同步，零依赖）
  static checkBasic(description: string): { pass: boolean; reason: string } {
    // 1. 敏感词
    for (const word of this.SENSITIVE_WORDS) {
      if (description.includes(word)) {
        return { pass: false, reason: `描述包含违规内容，请修改后重新提交` };
      }
    }
    // 2. 垃圾内容
    for (const pattern of this.SPAM_PATTERNS) {
      if (pattern.test(description)) {
        return { pass: false, reason: '故障描述无效，请填写真实故障信息' };
      }
    }
    return { pass: true, reason: '' };
  }

  // ========== 第二层：DeepSeek AI 语义审核 ==========

  // 调用 DeepSeek chat/completions API（OpenAI 兼容格式）
  // env: DEEPSEEK_API_KEY, DEEPSEEK_BASE_URL, DEEPSEEK_MODEL, DEEPSEEK_TIMEOUT
  static async checkAI(
    name: string, phone: string, lampId: string, description: string
  ): Promise<{ pass: boolean; reason: string; rawResponse?: string }> {
    // 1. 如果未配置 API Key，跳过 AI 审核（返回通过 + 标记不确定）
    // 2. 用 Node 18+ 原生 fetch 调用 DeepSeek
    // 3. 系统提示词定义审核规则
    // 4. 要求返回 JSON: { "pass": true/false, "reason": "..." }
    // 5. 异常处理：超时/网络错误 → 返回 pass=true, reason="AI服务暂时不可用"
  }

  // ========== 第三层：重复与频率检查（需要 DB） ==========

  static async checkDuplicate(
    phone: string, lampId: string
  ): Promise<{ pass: boolean; reason: string }> {
    // 同一手机号+路灯 24h 内已有记录 → 拒绝
  }

  static async checkRateLimit(phone: string): Promise<{ pass: boolean; reason: string }> {
    // 同一手机号当日上报 ≥5 次 → 拒绝
  }
}
```

**DeepSeek 提示词设计**：

```
你是一个智慧路灯系统的故障上报审核助手。请审核以下游客提交的故障上报内容是否合规。

审核规则：
1. 必须是真实的路灯故障描述（如：不亮、闪烁、灯杆倾斜、灯罩破损、线路外露等）
2. 不得包含广告、推销、联系方式
3. 不得包含政治敏感、色情、暴力内容
4. 不得是无意义的乱码或灌水文字
5. 如果是描述路灯故障，即使表达不够专业（如"灯坏了""不亮了"），也应判定通过

请严格以 JSON 格式回复：
{ "pass": true/false, "reason": "审核说明（不通过时说明具体原因）" }

---上报内容---
上报人：${name}
故障路灯：${lampId}
故障描述：${description}
```

### 3.3 `task3-backend/src/controllers/report.controller.ts`

**改造 `submitReport` 方法**：

```

```

② ComplianceCheckService.checkBasic(description)

③ ComplianceCheckService.checkDuplicate(phone, lampId)
   → 重复 → write report_audit_log → 返回 4001

   → audit_pass=0 → 返回 4001 + AI 驳回原因
   → audit_pass=1/2 → 返回 "上报已提交，等待审核"

## 四、错误码

| 错误码 | 含义 | 触发条件 |
|--------|------|---------|
| `4001` | 合规校验驳回 | 代码规则或 AI 判定不合规 |
| `4002` | 重复上报 | 同一手机+路灯 24h 内已提交 |
| `4003` | 上报频率超限 | 同一手机号当日 ≥5 次 |
| `5001` | AI 服务异常 | DeepSeek API 不可用（此时自动降级为人工审核，对游客透明） |

## 五、事务要求

`PUT /api/review/:id/approve` 必须保证原子性：

```
BEGIN TRANSACTION
  1. UPDATE report_audit_log SET review_status='approved', review_action='approved', reviewer=?, review_time=NOW()
  2. INSERT INTO alarms (device_id, device_name, alarm_type='fault_report', ...)
  3. INSERT INTO fault_reports (alarm_id=上一步ID, reporter_name, ...)
COMMIT
```

使用 `pool.getConnection()` + `conn.beginTransaction()` / `conn.commit()` / `conn.rollback()`。

## 六、依赖说明

**不需要新增任何 npm 依赖。** DeepSeek API 走 OpenAI 兼容格式，用 Node 18 内置的 `fetch` 直接调用即可。

```typescript
// 原生 fetch 调用 DeepSeek（无需 axios）
const response = await fetch(`${DEEPSEEK_BASE_URL}/v1/chat/completions`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${DEEPSEEK_API_KEY}`,
  },
  body: JSON.stringify({
    model: DEEPSEEK_MODEL,
    messages: [
      { role: 'system', content: SYSTEM_PROMPT },
      { role: 'user', content: `上报人：${name}\n故障路灯：${lampId}\n故障描述：${description}` },
    ],
    temperature: 0.1,    // 低温度，追求一致性判定
    max_tokens: 256,
  }),
  signal: AbortSignal.timeout(DEEPSEEK_TIMEOUT),
});
```
