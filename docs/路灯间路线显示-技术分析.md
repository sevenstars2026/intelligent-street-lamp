# 路灯间路线显示 — 技术可行性分析

## 1. 需求简述

在地图上展示路灯位置之间的游览路线（固定预定义路线），用户点击路灯可查看经过该路灯的路线，路线以彩色折线形式覆盖在地图上。

---

## 2. 当前技术栈

| 层 | 技术 |
|----|------|
| 地图引擎 | Leaflet v1.9.4 |
| 地图瓦片 | 高德地图（`webrd0{s}.is.autonavi.com`，style=8 无标注版） |
| 前端框架 | Vue 3 + Vite |
| 路由绘制 | Leaflet `L.polyline()` |
| 路径来源（三级回退） | ① 高德步行路径规划 API → ② OSRM 国际服务 → ③ 本地预定义途经点 |

---

## 3. 现有实现分析

### 3.1 三条预定义路线（已实现）

| 路线 ID | 名称 | 起止路灯 | 距离 | 途经点数量 |
|---------|------|----------|------|------------|
| 1 | 湖畔夜光步道 | lamp_001 → lamp_003 | 1.8km | 11 个 |
| 2 | 花海漫步路线 | lamp_001 → lamp_002 | 1.2km | 10 个 |
| 3 | 森林探幽小径 | lamp_002 → lamp_003 | 2.5km | 9 个 |

路线数据定义在两处：
- **业务数据**：[`useScenic.js`](tourist-frontend/src/composables/useScenic.js#L12-L16) — 路线名称、时长、距离、起止路灯 ID
- **地图坐标**：[`ScenicMap.vue`](tourist-frontend/src/components/ScenicMap.vue#L39-L60) — `ROUTE_WAYPOINTS` 对象，每条路线对应一个经纬度数组（沿真实道路曲线）

### 3.2 三级路径获取策略（已实现）

```
用户加载页面
    │
    ▼
renderRoutes() 遍历所有路线
    │
    ├─ ① fetchAmapRoute(start, end)        ← 高德步行路径规划
    │      ├─ 成功 → 解码 polyline → 绘制
    │      └─ 失败 ↓
    │
    ├─ ② fetchOSRMRoute(start, end)         ← OSRM 国际服务
    │      ├─ 成功 → 提取 GeoJSON → 绘制
    │      └─ 失败 ↓
    │
    └─ ③ ROUTE_WAYPOINTS[routeId]           ← 本地预定义途经点
           └─ 兜底方案，离线也能显示
```

- **第①级（高德）**：需要高德 Web 服务 API Key，通过 Vite proxy 转发（`/amap-api` → `restapi.amap.com`），Key 不暴露到前端
- **第②级（OSRM）**：免费、无需 Key，但服务器在国外，国内延迟较高（约 2-5 秒）
- **第③级（本地兜底）**：离线可用、零延迟，但路线较粗略（10 个点左右），不够贴合真实道路

### 3.3 路线渲染（已实现）

- 使用 Leaflet `L.polyline(coords, options)` 绘制彩色折线
- 三条路线使用不同颜色：橙（`#f0a050`）、蓝（`#5b9bd5`）、绿（`#6aab70`）
- 路线支持 tooltip（显示路线名称）和 click 事件
- 路线根据筛选条件（全部/拍照点/活动/路线）显示或隐藏
- 结果缓存到 `routeCoordsCache`，同一条路线不会重复请求

## 5. 待优化项（可选）

以下不是必须的，但如果想要更好的体验可以考虑：

### 5.1 高德 API Key 配置

当前需要配置环境变量 `VITE_AMAP_KEY` 才能使用第①级（高德路径规划）。如果没有配置，会自动降级到 OSRM 或本地途经点。

**获取 Key 步骤**：
1. 注册[高德开放平台](https://console.amap.com/dev/key/app)
2. 创建应用 → 添加 Key → 勾选「Web服务」API
3. 在项目根目录创建 `.env` 文件：`VITE_AMAP_KEY=你的key`

> Vite proxy 已配置好（`/amap-api` → `restapi.amap.com`），Key 不会暴露到浏览器端。

### 5.2 路线途经点精度

当前 `ROUTE_WAYPOINTS` 的途经点数量较少（9-11 个），在高德 API 不可用时路线不够平滑。如果需要离线场景下更真实的路线，可以：
- 手动在高德地图上沿真实道路采点，增加到 20-30 个途经点
- 或使用专业工具（如 QGIS）导出道路 shapefile 提取坐标

### 5.3 新增路线

如需新增第 4 条路线（如在 lamp_001 和 lamp_002 之间新增一条），修改两处即可：

1. **`useScenic.js`** `ROUTE_MOCK` 数组 — 追加路线元数据（id、名称、时长、距离、lampIds）
2. **`ScenicMap.vue`** `ROUTE_WAYPOINTS` 对象 — 追加该路线的途经点坐标数组

### 5.4 移除未使用的依赖

`leaflet-routing-machine` 已在 `package.json` 中声明但代码中未使用。该库依赖 OSRM 服务且 UI 较老旧，与当前自定义渲染方案不兼容。如果确定不需要，可从 `package.json` 中移除以减小包体积。

> 注：该库的机器路由功能（turn-by-turn 导航指令）与当前项目需求不匹配——景区游览路线是美学和体验导向的 curated path，不是最短路径导航。

## 6. 数据流总结

```
useScenic.js (数据层)
  │  ROUTE_MOCK[3条] → loadScenicData() → API优先，失败用Mock
  │
  ▼
MapPage.vue (页面层)
  │  routes 通过 props 传入 ScenicMap
  │
  ▼
ScenicMap.vue (地图渲染层)
  │  renderRoutes() → loadRouteCoords()
  │    ├─ 缓存命中 → 直接用
  │    └─ 缓存未命中 → 高德API → OSRM → ROUTE_WAYPOINTS
  │
  ▼
L.polyline() 绘制到 Leaflet 地图上
```
